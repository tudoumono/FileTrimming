# チャンキング検討タスク整理

## 1. この文書の目的

- この文書は、チャンキング検討における設計判断の正本として扱う
- 前提条件、これまでに試した内容、現時点で分かっていることを記録として残し、その上で次に考えるべきことを整理する
- 議論の結果はこの文書のチェックリスト（§6）に決定事項として記録し、蓄積する
- `docs/ドキュメント処理パイプライン詳細設計書_v2.md` は設計ドラフトとして残し、本文書の議論結果が出揃った段階で再構成する
- `docs/拡張子別/*.md` は拡張子ごとの補足メモとして扱う

## 2. 固定前提

- Dify 実行環境は、完全オフラインの閉域網でセルフホストされる
- Dify のバージョンは `1.13.0`
- Dify の配備構成は `Windows host + WSL + Docker Compose`
- ドキュメントの整形、構造化、チャンキングは、別のオンライン設置可能 PC で事前実施する
- 前処理後の成果物をオフライン環境へ持ち込み、まずは `Dify UI` で手動取り込みする
- そのため、最終成果物は「Dify に取り込めること」「Dify UI で取り込めること」を最優先条件とする
- Dify の接続先 LLM は `Ollama` 経由のローカルモデル群を前提とする
- 回答モデルは `llama-3-elyza-8b` を利用している
- 埋め込みモデルは `embeddinggemma300m` を利用している
- 仮置きの Python 実装は一旦撤去しており、現時点では `docs/` 配下の文書を正本として議論を進める
- 内部ドキュメントそのものは共有できない前提とする
- そのため、今後の議論や検討では、口頭での説明、要約、匿名化した説明、または近い性質のサンプル文を使って進める
- `docs/ドキュメント処理パイプライン詳細設計書_v2.md` は設計ドラフトとして参照用に残す。議論結果が出揃った段階で本文書の決定事項を反映して再構成する
- `docs/拡張子別/*.md` は拡張子ごとの補足メモとして扱う

## 3. これまでに試したこと

### 3.1 目的

- まずは Dify に登録できる状態に持っていくことを優先した
- 文書ごとの細かな性質差はまだ吸収せず、全形式を同一の流れで前処理した

### 3.2 実施した流れ

1. 各種ドキュメントを `MarkItDown` で Markdown に変換した
2. OpenAI API を使い、Markdown 内のノイズ除去、ゴミ情報の削除、崩れた表の整形を行った
3. 整形後の内容をもう一度 OpenAI に渡し、可能な限り書き下した文章へ変換した
4. Dify 側で改行を使って分割されることを意識し、1文ずつ扱いやすい形へ寄せた

### 3.3 この方法で意図していたこと

- 形式差をいったん無視し、Dify に入れやすい形へそろえる
- 表や箇条書きを、そのままではなく検索しやすい文章へ寄せる
- まずは「Dify で使えるか」を確認するためのベースラインを作る

### 3.4 具体例

- Excel にエラーコード表がある場合は、各行の内容を
- `エラーコード〇〇はこういう意図のエラーです`
- `このエラーの時にはこういう対処をしてください`
- `こういう条件で判定しています`
- のような文章へ書き換えた

## 4. 現時点で分かっていること

- 一律処理だけでは品質を十分に考慮できない
- Excel 系ではある程度機能しても、`Docs / Docx`、特に `RTF / TRF`、`BAGLES` ではうまくいかないケースが多い
- 「整形して自然文にすること」と「どの単位で区切るか」は、本来は別の論点である
- 今回の一律処理は、最終版の戦略ではなく、まず Dify に登録するための暫定ベースラインとして扱うのが妥当
- 今後は、形式ごとに別の処理方針を考える必要がある

## 5. 今回の議論で守る条件

### 5.1 ベストの定義

ここでいう「ベスト」は、まず次の 2 条件を満たすものとする。

1. Dify に取り込めるコンテンツであること
2. Dify UI で取り込めること

その上で次に考える。

- 検索で引きやすいか
- 元文書の意味を壊しにくいか
- 運用しやすいか

### 5.2 今はまだ決めなくてよいこと

- 最終的な自動化方式
- Knowledge API 前提の投入設計
- 形式横断で完全に統一されたチャンキング方式
- 厳密なトークン上限や細かな実装パラメータ

## 5.3 現時点で追加確認できた前提

- Dify に投入する最終ファイル形式は、現時点では `Markdown (.md)` を第一候補とする
- 今回の主眼は「事前に完全なチャンク成果物を作って渡す」ことではなく、「Dify に渡した後、Dify 標準のチャンキングがうまく機能する形にファイルを整える」ことにある
- したがって、現時点では `1チャンク = 1ファイル` を前提にせず、`1ドキュメントを Dify に渡し、その中身を Dify 側で分割させる` 方向を基本認識とする
- 前処理の役割は、Dify が分割しやすく、かつ意味を壊しにくい構造へ文書を整形することに置く
- そのため、ここでいう「チャンキング検討」は、厳密には `Dify 標準チャンキングがうまく働く前処理形式の検討` を含む

## 5.4 LLM 活用の設計原則

- パイプライン全体は LLM なしでも完走できる設計とする。LLM は品質向上のブースターとして使い、必須依存にはしない
- LLM は「解釈・再構成」に使い、「抽出」には使わない
  - 抽出（見出し・段落・表・図形の取得）は python-docx / COM で確定的に行う。再現性とデバッグ容易性を優先する
  - 解釈・再構成（図形の説明文化、複雑な表の親子関係解釈）は LLM で品質を上げる
- LLM 呼び出し部分は共通インターフェースとし、バックエンド（OpenAI / ローカル LLM / なし）を切り替え可能にする
- 前処理はオンライン PC で実施するため OpenAI API を利用可能だが、最終成果物としてオンライン PC に依存する形にはしない
  - オフライン環境ではローカル LLM またはLLM なしで同じパイプラインが動くことを保証する
- LLM による品質チェック（元文書との比較）は、元文書が LLM に直接渡せない形式（.docx 等）のため採用しない。品質チェックは機械的なマーカー付与（`LOW_CONFIDENCE` 等）で行う

### LLM 活用箇所の一覧

| ステップ | LLM なし（ベースライン） | LLM あり（品質向上） |
|---------|----------------------|-------------------|
| 図形解釈 | テキスト抽出 + 接続関係を機械的に列挙 | 列挙結果を意味のある説明文に再構成 |
| 表→半構造化テキスト | テンプレート的にラベル付きテキスト化 | 複雑な表の暗黙的な親子関係を解釈 |

## 5.5 入力フォルダ構造の前提

- 入力ドキュメントは機能ごとにフォルダで管理されている。1 フォルダ内に他の機能のドキュメントは混じらない
- 1 フォルダ内に異なる拡張子（.doc, .docx, .rtf, .xls, .xlsx, .bik 等）が常に混在する
- フォルダ内のファイルの命名規則は定まっておらず、ファイル名から並び順は判断できない
- 1 フォルダ内には表紙・変更履歴・仕様本文などが含まれるが、現時点では各ファイルを個別に Dify 1 document として扱う（フォルダ単位の結合は将来拡張: §10.6-7）
- パイプラインはフォルダ単位でスキャンし、フォルダ内の各ファイルを拡張子に応じた抽出器で処理する

## 5.6 ディレクトリ構成とパイプラインステップ

### ディレクトリ構成

```

  機能A/
    表紙.doc
    変更履歴.rtf
    仕様書.docx
    エラー一覧.xls
  機能B/
    ...

intermediate/                    # 中間成果物（ステップ別）
  01_normalized/                 # Step1: 正規化済み .docx
    機能A/
      表紙.docx
      変更履歴.docx
      仕様書.docx
    normalize_log.jsonl
  02_extracted/                  # Step2: 構造抽出 → 中間表現 JSON
    機能A/
      表紙.json
      変更履歴.json
      仕様書.json
    extract_log.jsonl
  03_transformed/                # Step3: 中間表現 → Markdown + 15MB超分割
    機能A/
      表紙.md
      変更履歴.md
      仕様書.md
    transform_log.jsonl

output/                          # 最終成果物（input のフォルダ構造をミラー）
  機能A/
    表紙.md
    変更履歴.md
    仕様書.md
  機能B/
    ...

src/
  extractors/                    # 拡張子別の抽出器
    word.py                      # .doc/.docx/.rtf → 中間表現
    excel.py                     # .xls/.xlsx（将来）
    bagles.py                    # .bik/.bca/.bci/.bpg（将来）
    text.py                      # .txt/.csv/.tsv（将来）

  llm/

    noop_backend.py              # LLM なしフォールバック
  models/
    intermediate.py              # 中間表現データクラス（全形式共通）
    metadata.py                  # 追跡情報
  pipeline/

    splitter.py                  # 15MB 超の個別ファイル分割

    to_markdown.py               # 中間表現 → 半構造化テキスト Markdown

  test_extract.py
```

### パイプラインステップと再実行

- 各ステップは前のステップの出力だけを入力とする。`input/` を直接参照するのは Step1 のみ
- 各ステップのログファイル（`*_log.jsonl`）で処理結果（成功/失敗/スキップ/警告）を追跡可能にする
- 任意のステップから再実行できる設計とする（例: LLM バックエンドを変えて Step3 だけやり直す）

| ステップ | 入力 | 出力 | 再実行時 |
|---------|------|------|---------|
| Step1: 正規化 | `input/` | `intermediate/01_normalized/` | input から再実行 |
| Step2: 構造抽出 | `01_normalized/` | `intermediate/02_extracted/` | 01 から再実行。正規化やり直し不要 |
| Step3: 変換+分割 | `02_extracted/` | `intermediate/03_transformed/` → `output/` | 02 から再実行。抽出やり直し不要 |

### 設計上の補足

- `output/` は `input/` のフォルダ構造をミラーする
- `intermediate/` はステップごとにフォルダ構造が変わっても問題ない。後から追えることが重要
- 未対応拡張子は Step1 でスキップされ、ログに記録される
- エラー発生時はスキップしてログに記録し、次のファイルに進む
- COM はシングルスレッド（STA）制約があるため、Step1 の正規化は逐次処理。性能問題が出た場合は実装段階で対策を検討する
- `input/` 内の実データは `.gitignore` で除外する（内部ドキュメントは共有不可の前提）

## 6. 今後優先的に取り組むべき作業

この節は、今後の議論結果を反映するための運用用チェックリストとして扱う。
各項目は、議論して結論が出た時点で ` [x] ` に変更し、その直下に `決定事項` を追記する。

- [x] Dify UI の取り込み制約を確認する
  - 決定事項:
    - 最終投入形式は Markdown (.md) で確定
    - Dify のファイルサイズ上限は 15MB。100MB のファイルが 1 つあり物理分割が必要。他は 15MB 以内の見込み
    - Dify 自動チャンキングを使用。改行・段落区切り・見出しを手がかりに分割される前提
    - 見出しは Dify への区切りヒントとして使うが、見出し = 1 チャンクとは限らない。目標は「1 見出し配下 = 1 つの意味まとまりで、できれば 1 チャンク以内に収まるサイズ」
    - 意味のまとまりの基本単位は「1 つの表＋その前後の説明文」
    - 表は Markdown テーブルではなく「項目ラベル付き半構造化テキスト」に変換して渡す。行列の意味や制約・対応関係を壊さないことを優先
    - 物理分割は 15MB 超だけでなく、意味的に大きすぎる文書も対象。15MB は UI のハード制約、意味単位での分割は運用・検索品質の判断軸として別に持つ
    - 見出しの付与・保持方針は形式別検討で個別に決める
- [x] `Docs / Docx / Word` の代表サンプル文書を選ぶ
  - 決定事項:
    - Word 系は全て設計書・仕様書。1機能が表紙・変更履歴・仕様本文の別ファイルに分かれている
    - 変更履歴だけで 3〜4 ファイルに分かれるケースもある
    - 最重要は仕様本文だが、表紙・変更履歴・仕様本文の自動判別は現状できない
    - 全ファイル同じ扱いで入れる（C案）を採用。2TB 規模で手動分類は非現実的
    - 代表サンプルは特定ファイルではなく、仕様本文の典型構造をベースに方針を決める
- [x] Word 系で、文書の性質と壊してはいけない情報を整理する
  - 決定事項:
    - 壊してはいけない情報: 見出し階層、表の行列関係、階層化された表の親子関係、ワークフロー図の要素間関係（順序・分岐）、画面レイアウトのフィールド定義、プログラム番号・識別子、制約事項の記述、元ファイル名の追跡性
    - 仕様本文の構造は対象により多様: 見出し→説明→表の繰り返し、ワークフロー図、画面レイアウト、プログラム番号参照、階層化された表
- [x] Word 系で、最終出力を `自然文` `半構造化テキスト` `表保持` のどれに寄せるか決める
  - 決定事項:
    - 半構造化テキスト（B案）で確定
    - 見出し階層は `##` `###` で保持
    - 表は項目ラベル付き半構造化テキストに変換（第 1 段階で決定済み）
    - 説明文はそのまま残す
    - ワークフロー図・画面レイアウトはテキスト説明に変換を試みる。復元困難時はフォールバックする
- [x] Word 系で、Dify UI に投入する 1 ファイルの単位を決める
  - 決定事項:
    - 原則: 元ファイル 1 つ = Dify 1 document
    - 例外: 15MB 超または意味的に大きすぎる場合は章・節単位で子文書化する
    - 例外分割時の最小ルール: parent_file, part_no, split_reason を子文書に持たせる
- [x] Word 系の暫定チャンキング戦略を決める
  - 決定事項:
    - 正規化: `.doc` → `.docx` に変換
    - 構造抽出: 見出し・段落・表・図形を抽出し中間表現を保持
    - 整形: 半構造化テキストへ変換。見出し保持、表はラベル付きテキスト化、図はテキスト説明に変換
    - フォールバック: 表・図の復元が困難な場合は `LOW_CONFIDENCE` `TABLE_FALLBACK` 等の出力マーカーを付与し「要確認」として残す
    - 追跡情報: 相対パス、元拡張子、元ファイル名を最低限必須とする。中間成果物には doc_role 推定用の材料（先頭見出し、定型語等）も保持する

    - 物理分割: 15MB 超または意味的に大きすぎる場合は章・節単位で分割。例外時は parent_file, part_no, split_reason を付与

    - `.trf` は誤記の可能性があるため設計対象から除外する

## 7. 確認すべき事実関係

### 7.1 Dify 側

- Dify UI が受け付けるファイル形式は何か → `.md` を含む主要形式に対応（§12.6 で確認済み）
- Markdown をそのまま取り込めるか → 取り込める。テキストとしてそのまま抽出される（§12.6.2）
- 1ファイルあたりの実運用上のサイズ上限はどの程度か → 15MB（環境変数で調整可能、§12.5 で確認済み）
- 細かく分割済みのファイルを入れた場合、Dify 側で再分割されるか → チャンク設定に従って再分割される。1チャンク以下でも document として登録される
- UI で手動投入する時、ファイル数が増えすぎると運用上どの程度厳しいか → デフォルトは 1 回 5 ファイル・15MB/ファイル。環境変数で調整可能（§12.5 で確認済み）
- YAML front matter はメタデータとして認識されるか → 認識されない。本文テキストとして扱われる（§12.6.1）
- メタデータはどう付与するか → アップロード後に UI またはAPI で後付け設定する（§12.6.3, §12.6.4）
- Markdown の見出しはチャンク境界になるか → なる。`MarkdownHeaderTextSplitter` で見出しごとに分割される（§12.6.2）
- 表やコードブロックの途中で切られるか → 切られる可能性がある。標準チャンカーには保護なし（§12.6.2）

補足:

- Dify 公式ドキュメントでは、各アップロード対象は `document` として扱われ、その後に `chunk` へ分割される
- したがって、現在の議論では「Dify に渡す 1 ファイル = 1 document」で考え、Dify 内部でどう chunk 化されるかを前提に整形方針を決める

### 7.2 文書側

- その形式の文書は、見出し中心か、表中心か、定義項目中心か
- 何を壊すと意味が失われるか
- 図、表、セルの対応関係、見出し構造などのうち、何を残す必要があるか
- どの単位なら「1つの意味のまとまり」と言えるか
- 自然文へ書き換えてよい形式か、構造を残した方がよい形式か

追加で分かっていること:

- 典型的な質問は、`エラーコードの意味`, `入力条件`, `その記述がどこに書かれているか` の確認である
- 影響調査として、`ここを修正したらどのドキュメントを修正すべきか`, `修正の影響範囲はどこか` を引きたい
- Word 系文書では、ワークフロー図の要素が多く、構造を読み違えるのが大きなリスクである
- 制約事項の誤読は避ける必要があり、単なる自然文化で意味を壊すことは避けたい
- 図や画像はテキスト説明に置き換えてよいが、その結果として文書構造が崩れるのは避けたい
- 表は可能なら表のまま残したいが、項目ラベル付きテキストへの変換も許容できる
- 一次文書との追跡性は最低限 `元ファイル名` が必須で、理想的には `ページ番号` や `表番号` まで追える状態が望ましい
- 更新運用は存在するが、現時点では手動での差し替えを前提とし、設計の最優先論点には置かない
- 文書は日本語中心だが、Bugles の識別番号、COBOL コード、エラーコード、プログラム番号などのコード要素を含む
- OCR が必要な文書は現時点では想定していない
- 最終評価基準は `Dify に登録できること` よりも `実際の質問で必要な情報が引けること` を上位に置く

### 7.3 Word 系を先に考える理由

- 実データの件数では `Docs / Docx / RTF / TRF` 系の比率が高い
- `RTF / TRF` は、現時点では Word へ変換してから扱う方針が有力である
- Word 系は、表、画像、テキストボックス、浮動オブジェクトなどが Markdown 化で崩れやすい
- そのため、Office 文書を直接 Markdown 化せず、先に構造を取る考え方を最初に検証する対象として適している

1. **メタデータ設計**: doc_role（cover/history/body）、revision_date、source_path、hash、group_id の定義。Knowledge API 移行時にまとめて設計する。ただし中間成果物には doc_role 推定材料を先行して保持する
2. **表の変換仕様**: 結合セル、空欄セルの継承、縦見出し、入れ子表、親子行の扱い。実サンプルで検証して決める。復元困難なケースはフォールバック対象とする
3. **図のフォールバック規則**: ワークフロー図・画面レイアウトについて、どこまで自動変換しどこで「要確認」に落とすか。図形内テキスト + 座標関係 + 接続順まで抽出し、解釈しきれないものはマーカー付きで残す方針
4. **Dify 自動チャンキングの検証**: 実サンプルで表・制約事項の途中切断が起きないか検証する。問題が出た場合は見出し/表/図の境界で事前チャンクする方式への切り替えを検討する
5. **変更履歴の重み分け**: 全ファイル同じ扱いで入れるが、検索時に古い仕様が混ざるリスクがある。「同じ扱いで入れる」と「同じ重みで使う」は分けて考える。API 移行時に doc_role で切り分ける設計を想定する
6. **構造抽出の技術選定**: `.doc` → `.docx` 変換は COM でよいが、`.docx` 本文の構造抽出まで COM に寄せると 2TB 規模で遅くて不安定になるリスクがある。実装段階では変換は COM、抽出は OpenXML 系の分離を検討する
7. **フォルダ単位の結合投入**: 現在は「元ファイル 1 つ = Dify 1 document」だが、将来的には「1 機能フォルダ = Dify 1 document」への拡張を検討する。フォルダ内の全ファイルを連結して 1 つの Markdown にすることで、機能単位での情報の関連付けが強化される。ただし変更履歴ファイルが混入して古い仕様を引用するリスクがあるため、現時点では採用しない。実施時には結合順序の決定ルール、変更履歴の除外またはセクション分離の仕組みが必要

### 12.6 Dify Markdown インポートの詳細調査（2026-03-12 追加）

#### 12.6.1 YAML front matter

- Dify は YAML front matter を認識しない。本文テキストとして扱われ、チャンクにノイズとして混入する
- したがって最終 Markdown にメタデータを埋め込まない。追跡情報は中間表現 JSON で保持する

#### 12.6.2 Markdown チャンキングの挙動

- `MarkdownHeaderTextSplitter` により、見出し（`#`, `##` 等）でチャンク境界が切られる
- デフォルト区切り文字は `\n\n`（段落）と `\n`（改行）。区切り文字はチャンクから除去される
- 表やコードブロックの途中で切られる可能性がある（標準チャンカーには特別な保護なし）
- サードパーティの Markdown Text Chunker プラグインで表保護が可能
  - https://marketplace.dify.ai/plugin/biaoo/md-text-chunker
- 見出し階層の親子関係は保持されず、各見出しが独立チャンクになる（既知の制約）
  - https://github.com/langgenius/dify/issues/15043
  - https://github.com/langgenius/dify/issues/13132

#### 12.6.3 メタデータの管理

- メタデータはドキュメントのアップロードとは別に、後付けで設定する仕組み
- ビルトインフィールド: `document_name`, `uploader`, `upload_date`, `last_update_date`, `source`（デフォルト無効、有効化が必要）
- カスタムフィールド: string / number / time の 3 型で作成可能
- UI ではアップロード後にメタデータエディタで設定する。一括編集にも対応
  - https://docs.dify.ai/en/use-dify/knowledge/metadata

#### 12.6.4 Knowledge API でのメタデータ操作

- ドキュメント作成 API（`create_by_text`, `create-by-file`）にはメタデータパラメータがない
- メタデータは作成後に別エンドポイントで付与する:
  - `POST /v1/datasets/{dataset_id}/metadata` — フィールド作成
  - `POST /v1/datasets/{dataset_id}/documents/metadata` — ドキュメントへの値割り当て
  - `GET /v1/datasets/{dataset_id}/metadata` — フィールド一覧
- API でのメタデータ一括付与パラメータ対応は要望されたが未実装
  - https://github.com/langgenius/dify/issues/20649
  - https://docs.dify.ai/en/use-dify/knowledge/manage-knowledge/maintain-dataset-via-api

#### 12.6.5 前処理

- 複数スペースは単一スペースに圧縮される
- 連続改行は縮約される
- URL・メールアドレスの除去はオプション

#### 12.6.6 チャンキングモード

- General モード: 全チャンク共通設定（区切り文字 + 最大チャンク長）。オーバーラップ対応
- Parent-child モード: 子チャンクで精密マッチング、親チャンクでコンテキスト提供。親は「段落」または「全文書」モード

#### 12.6.7 この調査から得た設計判断

- 最終 Markdown に YAML front matter やメタデータを埋め込まない
- 追跡情報（source_path, source_ext 等）は中間表現 JSON にのみ保持する
- 見出し設計がチャンク境界を決定するため、見出しの粒度と配置が品質に直結する
- 表の途中切断リスクは §10.6-4 で実サンプル検証する。問題が出た場合はプラグイン導入または事前チャンク方式を検討する
- メタデータの自動付与は Knowledge API 移行時に設計する（§10.6-1）

  - 仕様書本体（変更履歴なし）: 18ファイル
  - 変更履歴のみ: 7ファイル
  - 仕様書+変更履歴の混在: 2ファイル
  - 不明（表なし等）: 4ファイル
- **運用方針**: パイプラインでの自動分岐は行わない。Dify への登録はユーザーが手動で行うため、変更履歴ファイルの除外はユーザー判断に委ねる。ただし変更履歴テーブルの検出ロジック自体は将来的な自動化の際に活用可能

#### 文書構造パターン

- 段落のテキスト量は少ない（平均486字/ファイル）
- 内容の大半は表と図形に含まれている
- 「段落→表」の繰り返しパターンが多い（見出し的段落の後に表が続く構造）
