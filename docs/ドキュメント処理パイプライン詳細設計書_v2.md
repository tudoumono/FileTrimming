## 1. BAGLES定義書とは何か（調査結果）

### 概要

**BAGLES**（Business Application Generator's Library for Extensive Support）は、1983年に富士通が提供を開始した大規模基幹系システム開発ツール。主に金融機関の勘定系システムで80件以上の導入実績がある。

**核心的な特徴**: 開発者が「日本語・表形式の業務仕様書（BAGLES定義書）」を記述すると、そこからCOBOLソースプログラムが100%自動生成される。つまり**BAGLES定義書はソースコードそのもの**であり、設計書とプログラムが完全に一致する。

### BAGLES定義書の種類

BAGLES定義書は、目的別に複数の定義書で構成される：

| 定義書名 | 内容 | 構造の特徴 |
|---------|------|-----------|
| **チェック更新定義書** | データ項目のチェック条件・更新条件・エラー時の対処を記述 | 表形式。項番／区分／項目／更新内容／チェック内容／エラーコード／条件1〜3 |
| **業務用語定義書** | 業務で使う用語と意味の対応を定義 | 表形式。項番／区分／用語名／内容 |
| **コード定義書** | 業務処理で使うコードを業務名称で定義 | 表形式。項番／区分／単純条件名／値 |
| **フォーマット定義書** | 電文・レコードのデータ領域構成を定義（COBOL PIC句相当） | 表形式。項番／区分／階層／項目／形式（例：９(３)、Ｓ９(１５)） |
| **ロジック定義書** | モジュール単位の処理ロジックを記述 | 表形式。項番／区分／処理／条件1〜 |
| **インタフェース定義書** | プログラム間のI/O定義 | 表形式。項番／区分／エントリモジュール／入出力項目 |
| **履歴定義書** | 修正番号ごとの変更履歴を管理 | 変更管理用 |

### BAGLES定義書のファイル形式

**重要**: BAGLES定義書は**Excel/Wordとは全く別の独自形式**。

- BAGLESⅡ（メインフレーム用）の定義書は汎用機固有の形式
- Open BAGLES（Windows/Linux/UNIX用）の定義書はEclipseベースのInterstage Apworksプラグインで編集
- 富士通のソフトウェアにプレビュー機能がある（＝テキストエディタでは正しく読めない可能性）
- BAGLESⅡからOpen BAGLESへの形式変換（移行）ツールが存在する
- 現時点の提供物では `.bik` 8件、`.bca` 1件、`.bci` 1件、`.bpg` 1件を BAGLES として扱う

### BAGLES定義書の処理戦略

独自形式であるため、以下の3つのアプローチを優先順位で検討する：

#### アプローチ1: プレビュー／エクスポート経由（★推奨）

富士通のBAGLESツール上で、定義書を**テキスト・PDF・HTMLなどにエクスポート**できる場合、そのエクスポート結果を処理対象にする。

```
[BAGLES定義書] → [富士通ツールでエクスポート] → [テキスト/PDF/HTML]
                                                      ↓
                                                 [パイプラインで処理]
```

**確認事項**:
- Open BAGLESのドキュメント出力機能（DF-COBDOC等）でWord/Excel/HTML/CSV出力が可能かどうか
- バッチ出力（コマンドラインからの一括エクスポート）に対応しているか
- エクスポート形式の選択肢は何があるか

#### アプローチ2: テキスト直接解析

定義書ファイルがテキストベース（Shift_JISやEBCDIC→変換済みテキスト）の場合、表形式の構造パターンを正規表現でパースする。

```python
def parse_bagles_definition(filepath: str) -> dict:
    """
    BAGLES定義書のテキストパース
    
    定義書は固定フォーマットの表形式であるため、
    パターンマッチで構造を抽出できる可能性がある。
    
    期待される構造:
    - ヘッダ行（定義書名、対象名）
    - カラムヘッダ（項番、区分、処理、条件1...）
    - データ行（10, 20, 30...の項番で固定間隔）
    """
    
    with open(filepath, 'r', encoding='shift_jis', errors='replace') as f:
        content = f.read()
    
    # 定義書種別の判定
    definition_type = detect_definition_type(content)
    
    PARSERS = {
        'チェック更新定義書': parse_check_update,
        '業務用語定義書':     parse_business_terms,
        'コード定義書':       parse_code_definition,
        'フォーマット定義書': parse_format_definition,
        'ロジック定義書':     parse_logic_definition,
        'インタフェース定義書': parse_interface_definition,
        '履歴定義書':         parse_history_definition,
    }
    
    parser = PARSERS.get(definition_type, parse_generic_table)
    return parser(content)
```

最終手段として、BAGLESツールで印刷プレビュー→PDF保存し、そのPDFを処理する。表形式なのでOCR＋テーブル抽出が必要になる。

## 2.1 Phase 2: フォーマット正規化（Win32 COM）

Windows環境＋Microsoft Officeインストール済みのため、**Win32 COM Automation**でレガシー形式を変換する。LibreOfficeではなくOffice自身を使うことで、変換精度が最も高く、レイアウト崩れを最小限に抑えられる。

```python
import win32com.client
from pathlib import Path

def phase2_normalize_formats(classified: dict, work_dir: str) -> dict:
    """レガシー形式を現行形式にWin32 COMで変換"""
    
    # Word変換（.doc → .docx, .rtf → .docx）
    word_targets = (
        classified.get('word_legacy', []) +
        classified.get('rtf', [])
    )
    if word_targets:
        convert_with_word(word_targets)
    
    # Excel変換（.xls → .xlsx）
    excel_targets = classified.get('excel_legacy', [])
    if excel_targets:
        convert_with_excel(excel_targets)
    
    # PowerPoint変換（.ppt → .pptx）
    ppt_targets = classified.get('pptx_legacy', [])
    if ppt_targets:
        convert_with_powerpoint(ppt_targets)
    
    return reclassify(work_dir)


def convert_with_word(filepaths: list[str]):
    """Word COM で .doc/.rtf → .docx に変換"""
    word = win32com.client.Dispatch("Word.Application")
    word.Visible = False
    word.DisplayAlerts = False
    
    try:
        for filepath in filepaths:
            p = Path(filepath)
            output = p.with_suffix('.docx')
            
            doc = word.Documents.Open(str(p.resolve()))
            # 16 = wdFormatXMLDocument (.docx)
            doc.SaveAs2(str(output.resolve()), FileFormat=16)
            doc.Close(SaveChanges=False)
    finally:
        word.Quit()


def convert_with_excel(filepaths: list[str]):
    """Excel COM で .xls → .xlsx に変換"""
    excel = win32com.client.Dispatch("Excel.Application")
    excel.Visible = False
    excel.DisplayAlerts = False
    
    try:
        for filepath in filepaths:
            p = Path(filepath)
            output = p.with_suffix('.xlsx')
            
            wb = excel.Workbooks.Open(str(p.resolve()))
            # 51 = xlOpenXMLWorkbook (.xlsx)
            wb.SaveAs(str(output.resolve()), FileFormat=51)
            wb.Close(SaveChanges=False)
    finally:
        excel.Quit()


def convert_with_powerpoint(filepaths: list[str]):
    """PowerPoint COM で .ppt → .pptx に変換"""
    ppt = win32com.client.Dispatch("PowerPoint.Application")
    
    try:
        for filepath in filepaths:
            p = Path(filepath)
            output = p.with_suffix('.pptx')
            
            presentation = ppt.Presentations.Open(
                str(p.resolve()), WithWindow=False
            )
            # 24 = ppSaveAsOpenXMLPresentation (.pptx)
            presentation.SaveAs(str(output.resolve()), FileFormat=24)
            presentation.Close()
    finally:
        ppt.Quit()
```

**Win32 COM を選択した理由**:

| 観点 | Win32 COM (Microsoft Office) | LibreOffice (headless) |
|------|------------------------------|----------------------|
| 変換精度 | ★ Office自身による処理で最も正確 | 一部レイアウト崩れの可能性 |
| 環境依存 | Windows + Office 必須 | クロスプラットフォーム |
| 追加インストール | 不要（既存Office利用） | LibreOffice別途インストール必要 |
| xlwingsとの親和性 | ★ 同じCOM基盤で統一 | 別系統 |
| 将来のLinux移植 | 不可 | ★ そのまま動作 |

本パイプラインはPhase 4でxlwingsを使用しておりWindows依存が前提のため、Win32 COMで統一する。

## 3. ファイル形式別の処理設計

### 3.1 Excel（コード一覧・マスタ系データ）

**特徴**: 業務コード一覧、ファイル一覧、エラーコード対応表など。データは構造化されているが巨大。

**Windows環境のため xlwings をフル活用**:

```python
import xlwings as xw

def convert_excel_code_list(filepath: str) -> str:
    """
    コード一覧系Excelのマークダウン変換
    
    特徴:
    - 1シートが1つの対応表であることが多い
    - ヘッダ行 + データ行の典型的テーブル構造
    - 結合セルは少ない（一覧表なので）
    - ただし行数が数千〜数万になることがある
    
    xlwingsを使う理由:
    - Windowsなので直接Excel COMが使える
    - 数式の計算結果を正確に取得（openpyxl data_only=Trueより信頼性が高い）
    - オートフィルタの状態、条件付き書式の評価結果も取得可能
    """
    
    app = xw.App(visible=False)
    try:
        wb = app.books.open(filepath)
        md_parts = []
        
        for sheet in wb.sheets:
            md_parts.append(f"# シート: {sheet.name}\n")
            
            used = sheet.used_range
            if used is None or used.value is None:
                md_parts.append("（空のシート）\n")
                continue
            
            values = used.value
            if not isinstance(values, list):
                values = [[values]]
            elif not isinstance(values[0], list):
                values = [values]
            
            # ヘッダ行の推定（1行目をヘッダとする）
            headers = [str(v) if v is not None else '' for v in values[0]]
            
            # Markdownテーブル生成
            md_parts.append('| ' + ' | '.join(headers) + ' |')
            md_parts.append('| ' + ' | '.join(['---'] * len(headers)) + ' |')
            
            for row in values[1:]:
                cells = [str(v) if v is not None else '' for v in row]
                # 改行を含むセルの処理
                cells = [c.replace('\n', ' / ') for c in cells]
                md_parts.append('| ' + ' | '.join(cells) + ' |')
            
            md_parts.append('')
        
        wb.close()
        return '\n'.join(md_parts)
    
    finally:
        app.quit()
```

**Excelのチャンキング戦略**: コード一覧はシート単位がそのまま意味のあるまとまり。1シートが巨大な場合は「コード体系の区切り」（例: エラーコードの頭文字ごと）で分割するのが理想だが、ルールベースでは困難なため行数で等分割＋オーバーラップとする。

### 3.2 Word（詳細設計書・文章＋図表）

**特徴**: 処理条件や業務ロジックが文章で記述。図（フロー図、概念図）や表（条件分岐表、I/F一覧）が混在。

```python
from markitdown import MarkItDown

def convert_word_design_doc(filepath: str) -> str:
    """
    Word設計書のMarkdown変換
    
    MarkItDownを使う理由:
    - 見出し構造（Heading1〜4）をMarkdown見出しに正しくマッピング
    - 表（テーブル）をMarkdownテーブルに変換
    - リストの階層構造を保持
    
    課題: Word内の図（画像）の扱い
    - MarkItDownはインライン画像をBase64で埋め込む場合がある
    - RAG用途では画像の内容説明（ALTテキスト）が重要
    - フロー図等はLLMで説明文を生成する必要がある場合も
    """
    
    mid = MarkItDown()
    result = mid.convert(filepath)
    md_content = result.text_content
    
    # Word内の図に対する後処理
    md_content = handle_embedded_images(md_content, filepath)
    
    return md_content


def handle_embedded_images(md_content: str, original_path: str) -> str:
    """
    Word内の図への対処
    
    戦略:
    1. 図のBase64埋め込みを検出
    2. 図ごとにプレースホルダーに置換: [図: {連番}]
    3. 図が重要（フロー図等）な場合はLLMで説明生成（Phase 5で実施）
    4. 単なる装飾画像の場合は除去
    """
    import re
    
    # Base64画像パターンの検出
    img_pattern = r'!\[([^\]]*)\]\(data:image/[^)]+\)'
    
    counter = [0]
    def replace_image(match):
        counter[0] += 1
        alt_text = match.group(1)
        if alt_text:
            return f"[図{counter[0]}: {alt_text}]"
        else:
            return f"[図{counter[0]}]"
    
    return re.sub(img_pattern, replace_image, md_content)
```

**Wordのチャンキング戦略**: Heading単位で分割。日本語の設計書はHeading1が「章」、Heading2が「節」に対応することが多い。Heading2単位での分割を基本とし、トークン超過時にはさらにHeading3で分割。

```python
        'チェック更新定義書',
        '業務用語定義書',
        'コード定義書',
        'フォーマット定義書',
        'ロジック定義書',
        'インタフェース定義書',
        '履歴定義書',
        '入出力定義書',
        'マクロ定義書',
    ]
    
    pattern = '|'.join(re.escape(kw) for kw in DEFINITION_KEYWORDS)
    
    # 定義書名の出現位置で分割
    positions = [m.start() for m in re.finditer(pattern, content)]
    
    blocks = []
    for i, pos in enumerate(positions):
        end = positions[i + 1] if i + 1 < len(positions) else len(content)
        blocks.append(content[pos:end].strip())
    
    return blocks


def bagles_definitions_to_markdown(definitions: list[BaglesDefinition]) -> str:
    """
    パース済みBAGLES定義書をMarkdownに変換
    
    変換方針:
    - 各定義書を##セクションとして出力
    - 表形式のデータはMarkdownテーブルで表現
    - COBOL形式の型情報（９(３)、Ｓ９(１５)等）はそのまま保持
    - 業務用語の「〔〕」参照はリンク的に保持
    - 条件式はコードブロックとして出力
    """
    
    md_parts = []
    
    for defn in definitions:
        md_parts.append(f"## {defn.definition_type}")
        
        if defn.title:
            md_parts.append(f"**対象**: {defn.title}\n")
        
        # メタデータ
        for k, v in defn.metadata.items():
            md_parts.append(f"- {k}: {v}")
        
        if defn.metadata:
            md_parts.append('')
        
        # テーブル形式で出力
        if defn.columns and defn.rows:
            md_parts.append(
                '| ' + ' | '.join(defn.columns) + ' |'
            )
            md_parts.append(
                '| ' + ' | '.join(['---'] * len(defn.columns)) + ' |'
            )
            
            for row in defn.rows:
                cells = [
                    str(row.get(col, '')).replace('\n', ' ')
                    for col in defn.columns
                ]
                md_parts.append('| ' + ' | '.join(cells) + ' |')
        
        md_parts.append('')
    
    return '\n'.join(md_parts)
```

## 5. Phase 5: 品質判定 & 構造化（ハイブリッド方式）

### ファイル形式ごとの品質判定基準

```python
def assess_quality(md_content: str, file_type: str) -> str:
    """
    品質スコアを返す: 'HIGH', 'MEDIUM', 'LOW'
    
    ファイル形式ごとに判定基準が異なる:
    
    Excel（コード一覧）:
      - テーブルの列数が揃っている → HIGH
      - 結合セルによる列ずれがある → MEDIUM
      - テーブル検出自体に失敗     → LOW
    
    Word（設計書）:
      - 見出し構造あり＋テーブル正常 → HIGH
      - 図のプレースホルダが多い     → MEDIUM（図の説明生成が必要）
      - 文字化け・レイアウト崩壊     → LOW
    
    BAGLES:
      - 定義書パーサが正常動作       → HIGH（構造化済みなので）
      - パーサ部分失敗               → MEDIUM
      - パーサ完全失敗（テキスト状態）→ LOW
    
    PDF:
      - テキスト抽出成功＋構造あり   → HIGH
      - テキスト抽出成功＋構造なし   → MEDIUM
      - スキャンPDF（OCR必要）       → LOW
    """
    
    score = 100
    
    # 共通チェック
    if detect_garbage_ratio(md_content) > 0.1:
        score -= 40
    if not md_content.strip():
        return 'LOW'
    
    # ファイル形式別チェック
    if file_type == 'excel':
        if not has_valid_tables(md_content):
            score -= 30
        if has_column_misalignment(md_content):
            score -= 20
    
    elif file_type == 'word':
        image_placeholder_count = md_content.count('[図')
        if image_placeholder_count > 5:
            score -= 15  # 図が多いと説明が必要
        if not has_heading_structure(md_content):
            score -= 20
    
    elif file_type == 'bagles':
        # BAGLESは定義書パーサが動けばほぼHIGH
        if '## チェック更新定義書' in md_content or \
           '## ロジック定義書' in md_content:
            score += 10  # パーサ成功ボーナス
    
    if score >= 70:
        return 'HIGH'
    elif score >= 40:
        return 'MEDIUM'
    else:
        return 'LOW'
```

### Difyナレッジベースへの出力

```python
def write_dify_output(chunks: list[dict], output_dir: str, doc_id: str):
    """
    Dify Knowledge Base に投入する形式で出力
    
    Difyの想定:
    - ファイル単位でアップロード（.md or .txt）
    - Dify側でも自動チャンキングが行われる
    - ただし、事前にセマンティックに分割しておく方が精度が高い
    
    出力戦略:
    - 1チャンク = 1ファイル（.md）
    - ファイル名にメタデータを埋め込む
    - 各ファイルの先頭にYAML Front Matterでメタデータ付与
    """
    
    doc_dir = Path(output_dir) / doc_id
    doc_dir.mkdir(parents=True, exist_ok=True)
    
    for chunk in chunks:
        idx = chunk['metadata']['chunk_index']
        filename = f"{doc_id}_chunk{idx:03d}.md"
        
        # YAML Front Matter
        front_matter = build_front_matter(chunk['metadata'])
        
        content = f"---\n{front_matter}---\n\n{chunk['content']}"
        
        (doc_dir / filename).write_text(content, encoding='utf-8')


def build_front_matter(metadata: dict) -> str:
    """Dify向けYAML Front Matter"""
    lines = []
    lines.append(f"source_file: {metadata.get('source_file', '')}")
    lines.append(f"file_type: {metadata.get('file_type', '')}")
    lines.append(f"chunk_index: {metadata.get('chunk_index', 0)}")
    lines.append(f"total_chunks: {metadata.get('total_chunks', 1)}")
    
    if metadata.get('sheet_name'):
        lines.append(f"sheet_name: {metadata['sheet_name']}")
    if metadata.get('heading_path'):
        lines.append(f"section: {metadata['heading_path']}")
    if metadata.get('definition_type'):
        lines.append(f"bagles_type: {metadata['definition_type']}")
    
    return '\n'.join(lines) + '\n'
```
