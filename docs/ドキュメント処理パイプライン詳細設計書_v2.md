## 0. 今回の見直しポイント

- OpenAI API で利用するモデル名の列挙が古くなっていたため、設計書上は「固定モデル名」ではなく「設定で切り替える前提」に整理した

- Word/RTF の高度なレイアウト保持や BAGLES の高度パースは「将来拡張候補」と「現行実装」を分けて整理した
- 詳細設計書と拡張子別メモの役割分担を明文化し、現行実装の正本を本書へ統一した
- `docs/Task.md` で整理した前提条件と議論順を反映し、仮置き実装ではなく現時点の設計方針を優先する文書として見直す
- Dify UI での手動登録と Dify 標準チャンキングを前提に、Phase 6 の位置づけを「事前チャンク成果物の出力」から「Dify 投入向け document 整形」寄りに整理する

### 0.1 文書の役割分担

- 本書は、現時点の設計方針、Phase ごとの責務、前提条件、優先順位の正本とする
- 仮置きの Python 実装は一旦撤去しており、本書の内容を正本として検討する
- 本書中の `steps/...` `utils/...` `llm/...` などのパスは、将来実装時の想定配置を示す
- `docs/Task.md` は議論メモと検討順整理に使い、本書へ反映すべき内容が固まったら本書を更新する

- 実装変更時は、まず本書を更新し、必要な場合のみ関連する拡張子別メモを追随更新する

#### 1.1.1 Dify 配備・運用制約

- Dify 実行環境そのものは完全オフラインであり、外部 API や外部モデル配信先へ直接アクセスしない前提とする
- 入力ドキュメントの整形、Markdown 化、品質補正、チャンキングは、Dify 実行環境とは別の前処理 PC で完了させる
- 前処理後の成果物を閉域網へ搬入し、Dify UI からナレッジ登録する運用を当面の標準とする
- このため、Phase 6 の出力はプログラム連携しやすさよりも、Dify UI で受け付けられる形式・サイズ・ファイル数であることを優先する
- Dify 側の検索・応答で使うモデルは `Ollama` 配下のローカルモデルを前提とし、GPT OSS 系を含む複数候補へ差し替えられる構成を想定する
- 内部ドキュメント原本は共有できない前提のため、検討やレビューは口頭説明、要約、匿名化した説明、近い性質のサンプル文を使って進める
- Dify へ渡す単位は、当面 `1ファイル = 1 document` を基本とし、Dify 側の標準 chunking がうまく働くように事前整形する

### 1.2 LLM / OpenAI 方針

- 現行コードは `llm/factory.py` と `llm/openai_client.py` により、Chat Completions 互換の呼び出しで OpenAI API を利用する
- この `openai` / `ollama` 切り替えは、前処理パイプライン側の構造化補正用 LLM を指し、Dify 実行環境側の接続先とは分けて考える
- OpenAI API 側の利用可能モデルは更新頻度が高いため、本設計書では `gpt-4o / gpt-4o-mini` のような固定列挙を中心にしない
- 2026-03-11 時点の OpenAI 公式ドキュメントでは GPT-5 系が前面に出ているが、本リポジトリの現行実装は `.env` の `OPENAI_MODEL` 差し替えで追従する前提とする
- Responses API への移行は未実施。現行仕様はあくまで Chat Completions 互換クライアント前提である

## 2. 全体アーキテクチャ

```text
[入力フォルダ]
    |
    v
 [オンライン前処理 PC]
    |
    v
Phase 1: コピー & ファイル分類
  - 入力フォルダ配下を step1/files へコピー
  - 拡張子ベースでファイル種別を分類
  - classification.json を出力

    |
    v
Phase 2: フォーマット正規化
  - .doc/.rtf/.trf -> .docx (Word COM)
  - .xls -> .xlsx (Excel COM)
  - .ppt -> .pptx (PowerPoint COM)
  - その他はパススルー
  - normalization_log.json を出力

    |
    v
Phase 3: トークン推定 & 物理分割
  - ファイル種別ごとに簡易テキスト抽出
  - `TOKEN_LIMIT` 超過時のみ行単位で分割
  - split_log.json を出力

    |
    v
Phase 4: Markdown 変換
  - Excel -> xlwings
  - Word/PDF/PPT -> MarkItDown
  - RTF/TRF -> Step 2 で `.docx` 化できた場合は Word ルート、失敗時のみ `pandoc` フォールバック
  - BAGLES -> 簡易パーサ
  - 図や画像は [図N] プレースホルダへ置換
  - conversion_log.json を出力

    |
    v
Phase 5: 品質判定 & 構造化
  - Markdown 品質スコアリング
  - HIGH/MEDIUM -> ルールベース補正

    |
    v
Phase 6: Dify 投入向け整形 & 出力
  - Dify が分割しやすい Markdown document へ整形

  - Dify UI 取り込み用ディレクトリを生成

    |
    v
[搬入用出力フォルダ]
    |
    v
```

## 3. Phase 別詳細設計

### 3.1 Phase 1: コピー & ファイル分類

実装対象:

- `steps/step1_copy.py`
- `utils/file_utils.py`

仕様:

- 入力フォルダ配下の全ファイルを、元のディレクトリ構造を保ったまま `work/step1/files` にコピーする
- 分類結果は `work/step1/classification.json` に `{種別: [相対パス...]}` 形式で保存する
- 現行実装は拡張子ベース分類が中心であり、magic bytes やヘッダ再判定は本流実装にはまだ入れていない

設計上の注意:

- 分類ルールは 1 箇所に集約する
- `.trf` を `trf` として扱う
- BAGLES の拡張子群は、提供実績ベースの `.bik/.bca/.bci/.bpg` と、旧実装由来の `.bgl/.bag/.def` の扱いを整理して統一する

### 3.2 Phase 2: フォーマット正規化

実装対象:

- `steps/step2_normalize.py`

正規化マッピング:

| 入力種別 | 変換先 | 実装 |
|----------|--------|------|
| `word_legacy` | `.docx` | Word COM |
| `rtf` | `.docx` | Word COM |
| `trf` | `.docx` | Word COM |
| `excel_legacy` | `.xlsx` | Excel COM |
| `pptx_legacy` | `.pptx` | PowerPoint COM |
| その他 | そのまま | コピー |

仕様:

- 変換成功時は `normalization_log.json` に `source` `source_type` `output` `output_type` `strategy` を記録する
- `.rtf/.trf` は `requires_visual_review=true` を付与し、図・フロー図・埋め込みオブジェクトの確認対象として残す
- 変換失敗時は原本コピーへフォールバックし、後段処理を継続する

設計判断:

- Windows + Office 前提のため、LibreOffice ではなく COM を採用する
- xlwings を採用している Excel ルートと整合が取りやすい

### 3.3 Phase 3: トークン推定 & 物理分割

実装対象:

- `steps/step3_split.py`
- `utils/token_utils.py`

現行仕様:

- `tiktoken.encoding_for_model("gpt-4o")` を使ってトークン推定する
- ここで使う `gpt-4o` は推定用エンコーダ指定であり、実際の推論モデル固定を意味しない
- `TOKEN_LIMIT` 以下、またはテキスト抽出結果が空の場合は分割せず原本を通過させる
- `TOKEN_LIMIT` 超過時のみ、行単位オーバーラップ分割を行い、分割後は `.txt` として保存する

簡易テキスト抽出ルート:

| 種別 | 抽出方法 |
|------|----------|
| `text` | `read_text_file()` |
| `excel` | `openpyxl` でセル値を列挙 |
| `word` | `python-docx` で段落を連結 |
| `pdf` | `PyMuPDF` でページテキストを抽出 |
| その他 | 文字列読み込みを試行し、失敗時は空文字 |

制約:

- 現行実装は「シート単位」「見出し単位」「定義書単位」のような意味分割は未実装
- 大きいファイルは最終的に行単位分割へ寄るため、将来的には Excel / Word / BAGLES の意味単位分割へ改善する

### 3.4 Phase 4: Markdown 変換 / 構造抽出

実装対象:

- `steps/step4_markdown.py`

変換ルート:

| 種別 | 変換方法 | 備考 |
|------|----------|------|
| `excel` | `xlwings` | `used_range` を Markdown テーブル化 |
| `word` | `MarkItDown` | `.docx` と、Step 2 で正規化済みの `.rtf/.trf` を含む |
| `pptx` | `MarkItDown` | PowerPoint |
| `pdf` | `MarkItDown` | PDF |
| `bagles` | 簡易パーサ | 定義書キーワードで区切ってコードブロック化 |
| `rtf` / `trf` | `pandoc` -> `MarkItDown` -> テキスト読み込み | Step 2 で `.docx` 化できなかった場合のフォールバック |
| `text` | そのまま読み込み | エンコーディングフォールバックあり |

共通後処理:

- Markdown / HTML 画像タグを検出し、`[図N]` または `[図N: alt]` プレースホルダへ置換する
- Step 2 の `normalization_log.json` を参照して `source_type` を引き継ぐ
- 変換結果は `conversion_log.json` に保存する
- `source_type` が `rtf` / `trf` の場合でも、正規化が成功していれば実際の変換器は `word -> MarkItDown` になる

設計上の位置づけ:

- [docx 取り扱いメモ](./拡張子別/docx.md) や [rtf 取り扱いメモ](./拡張子別/rtf.md) にある「Word COM + OOXML 抽出」「HTML table 保持」は将来拡張候補である
- 現時点の詳細設計書では、現行実装の主ルートを優先して記載する
- ただし、`Docs / Docx / Word` と `RTF / TRF` は優先検討対象であり、Speaker Deck `RAG x FINDY` の考え方を踏まえ、直接 Markdown へ潰す前に構造を取るルートを優先的に検討する
- 特に Word 系では、見出し、段落、表、画像、図形、テキストボックス、アンカー情報などを中間表現として保持してから最終 Markdown を再構成する方針を第一候補とする

### 3.5 Phase 5: 品質判定 & 構造化

実装対象:

- `steps/step5_structure.py`
- `llm/base.py`
- `llm/factory.py`
- `llm/openai_client.py`

品質判定ロジック:

- 共通: 制御文字比率、空文書、罫線ノイズを評価する
- Excel: Markdown テーブルの有無を重視する
- Word: 見出し構造と `[図N]` の数を見る
- BAGLES: `## チェック更新定義書` などのキーワードをボーナス判定に使う

処理方針:

| 品質 | 現行処理 |
|------|----------|
| HIGH | `rule_based_cleanup()` のみ |
| MEDIUM | `rule_based_cleanup()` のみ |
| LOW | LLM があれば再構造化、なければ `rule_based_cleanup()` |

LLM 呼び出し仕様:

- 呼び出し先は `LLMClient` 抽象へ隠蔽する
- `.env` の `LLM_PROVIDER` で `openai` / `ollama` を切り替える
- OpenAI 利用時は `OPENAI_MODEL` をそのまま利用する
- 現行既定値は `gpt-4o-mini` だが、設計としては任意の利用可能モデルへ差し替え可能とする

制約:

- Step 4 で元拡張子情報が一部失われるため、Step 5 はファイル名ベースで元種別を推定している
- MEDIUM 品質での「部分的 LLM 補正」は未実装

### 3.6 Phase 6: Dify 投入向け整形 & 出力

実装対象:

- `steps/step6_chunk.py`

設計方針:

- 当面は Dify `1.13.0` の UI で手動登録する前提のため、最終成果物は `Markdown (.md)` の document を基本とする
- ここでの主眼は「事前に完全なチャンク成果物を作って渡す」ことではなく、「Dify に渡した後、Dify 標準 chunking がうまく機能する形に整形する」ことに置く
- 基本単位は `1ファイル = 1 document` とし、Dify 側で chunk へ分割させる
- したがって、最終出力は `見出し構造`, `表`, `項目ラベル`, `出典追跡情報` を保ちつつ、Dify が自然に分割しやすい構造を優先する
- `1チャンク = 1ファイル` や YAML Front Matter 付きの事前チャンク成果物は、現行の仮実装に存在しても、現時点では確定仕様としない
- 追跡性として、最低限 `元ファイル名` を保持し、可能なら `ページ番号` `表番号` `節名` まで残す

Word 系の優先方針:

- `Docs / Docx / Word` を最優先の検討対象とする
- Word 系は、ワークフロー図、表、画像、テキストボックス、浮動オブジェクトで構造が崩れやすいため、直接 Markdown 化より `構造抽出 -> 中間表現 -> 再構成` を優先する
- 最終 Markdown は、完全な自然文への書き下しよりも、見出しや項目ラベルを残した形を優先候補とする

RTF / TRF の方針:

- `RTF / TRF` は、まず Word へ正規化して Word 系の方針へ合流させる
- 合流できない場合のみ、補助的なフォールバックを検討する

## 5. 実装上の課題と整理事項

1. BAGLES の拡張子判定がソース上で一元化されていない
2. `.trf` は設計対象に追加したが、実データでの変換品質確認は未実施
3. Phase 3 の分割はまだ意味分割ではなく、行単位分割が主である
4. Step 5 が元ファイル種別をファイル名推定しているため、判定の安定性が低い
5. Word / RTF の構造保持抽出は拡張子別メモにあるが、本流実装には未反映
6. OpenAI 側はモデル更新が早いため、設計書に特定モデル名をハードコードし続けない運用が必要
7. OpenAI の Responses API へ移行する場合は、現行の Chat Completions 互換実装との差分整理が必要
8. Dify UI 上で Markdown document がどのように chunk 化されるかを、実データに近いサンプルで確認する必要がある
9. 現時点のローカル実装は仮置きであり、Phase 6 の事前チャンク出力仕様は確定していない

---

## 6. 次のステップ

1. Dify UI の chunk 設定と、Markdown document 投入時の挙動を確認する
2. `Docs / Docx / Word` の代表サンプルを基に、壊してはいけない構造を整理する
3. Word 系の中間表現として何を保持するかを決める
4. `RTF / TRF` を Word へ正規化して同一方針へ合流できるかを検証する
5. その後に `Excel` のレコード / 表 / シート単位の扱いを整理する
6. 最後に `BAGLES` の定義書単位と項目単位の扱いを整理する
