## 0. 今回の見直しポイント

- OpenAI API で利用するモデル名の列挙が古くなっていたため、設計書上は「固定モデル名」ではなく「設定で切り替える前提」に整理した
- 対象拡張子として `.trf` を追加し、RTF 系の処理ルートを `.rtf/.trf` の両方で明記した
- Phase 3-6 の記述は、現時点の設計方針に合わせて簡潔化した
- Word/RTF の高度なレイアウト保持や BAGLES の高度パースは「将来拡張候補」と「現行実装」を分けて整理した
- 詳細設計書と拡張子別メモの役割分担を明文化し、現行実装の正本を本書へ統一した
- `docs/Task.md` で整理した前提条件と議論順を反映し、仮置き実装ではなく現時点の設計方針を優先する文書として見直す
- Dify UI での手動登録と Dify 標準チャンキングを前提に、Phase 6 の位置づけを「事前チャンク成果物の出力」から「Dify 投入向け document 整形」寄りに整理する

### 0.1 文書の役割分担

- 設計判断の正本は `docs/Task.md` とする。議論の経緯、決定事項、検討順はすべて Task.md に記録する
- 本書は設計ドラフト（参照用）として残す。Phase 構成、技術選定、処理フローなどの実装寄りの情報を保持する
- Task.md の議論結果が出揃った段階で、本書を Task.md の決定事項に合わせて再構成する
- `docs/拡張子別/*.md` は、拡張子固有の論点、品質リスク、試験観点、将来拡張候補を整理する補足メモとする

#### 1.1.1 Dify 配備・運用制約

- Dify 実行環境そのものは完全オフラインであり、外部 API や外部モデル配信先へ直接アクセスしない前提とする
- 入力ドキュメントの整形、Markdown 化、品質補正、Dify 標準 chunking が働きやすい形への事前整形は、Dify 実行環境とは別の前処理 PC で完了させる
- 前処理後の成果物を閉域網へ搬入し、Dify UI からナレッジ登録する運用を当面の標準とする
- このため、Phase 6 の出力はプログラム連携しやすさよりも、Dify UI で受け付けられる形式・サイズ・ファイル数であることを優先する
- Dify 側の検索・応答で使うモデルは `Ollama` 配下のローカルモデルを前提とし、GPT OSS 系を含む複数候補へ差し替えられる構成を想定する
- 内部ドキュメント原本は共有できない前提のため、検討やレビューは口頭説明、要約、匿名化した説明、近い性質のサンプル文を使って進める
- Dify へ渡す単位は、当面 `1ファイル = 1 document` を基本とし、Dify 側の標準 chunking がうまく働くように事前整形する
- Dify UI の単一ファイル上限は `15MB` を前提とし、これを超える文書は意味的に妥当な区切りで物理分割する
- Dify の自動チャンキングは、改行、段落区切り、見出しを手がかりに分割される前提で設計する

### 1.2 LLM / OpenAI 方針

- この `openai` / `ollama` 切り替えは、前処理パイプライン側の構造化補正用 LLM を指し、Dify 実行環境側の接続先とは分けて考える
- OpenAI API 側の利用可能モデルは更新頻度が高いため、本設計書では `gpt-4o / gpt-4o-mini` のような固定列挙を中心にしない
- 2026-03-11 時点の OpenAI 公式ドキュメントでは GPT-5 系が前面に出ているが、本リポジトリの現行実装は `.env` の `OPENAI_MODEL` 差し替えで追従する前提とする
- Responses API への移行は未実施。現行仕様はあくまで Chat Completions 互換クライアント前提である

## 3. Phase 別詳細設計

### 3.1 Phase 1: コピー & ファイル分類

- 入力フォルダ配下の全ファイルを、元のディレクトリ構造を保ったまま `work/step1/files` にコピーする
- 分類は拡張子ベースを基本とする

設計上の注意:

- 分類ルールは 1 箇所に集約する
- `.trf` を `trf` として扱う
- BAGLES の拡張子群は、提供実績ベースの `.bik/.bca/.bci/.bpg` と、旧実装由来の `.bgl/.bag/.def` の扱いを整理して統一する

### 3.2 Phase 2: フォーマット正規化

正規化マッピング:

| 入力種別 | 変換先 | 実装 |
|----------|--------|------|
| `word_legacy` | `.docx` | Word COM |
| `rtf` | `.docx` | Word COM |
| `trf` | `.docx` | Word COM |
| `excel_legacy` | `.xlsx` | Excel COM |
| `pptx_legacy` | `.pptx` | PowerPoint COM |
| その他 | そのまま | コピー |

- `.rtf/.trf` は `requires_visual_review=true` を付与し、図・フロー図・埋め込みオブジェクトの確認対象として残す
- 変換失敗時は原本コピーへフォールバックし、後段処理を継続する

設計判断:

- Windows + Office 前提のため、LibreOffice ではなく COM を採用する
- xlwings を採用している Excel ルートと整合が取りやすい

### 3.3 Phase 3: トークン推定 & 物理分割

- この段階では、後段処理や Dify 取り込みで扱いにくい大きさの文書を調整する
- `15MB` 超はハード制約として物理分割し、それ未満でも意味単位が大きすぎる場合は章・節単位などで分割候補を検討する
- 具体的な分割方式や閾値は形式ごとの議論結果に合わせて決める
- 現時点では、意味単位分割を優先して検討する

### 3.4 Phase 4: 構造抽出 / Markdown 化

変換ルート:

| 種別 | 変換方法 | 備考 |
|------|----------|------|
| `excel` | `xlwings` | 構造抽出起点とし、最終出力では Markdown テーブル固定にしない |
| `word` | `Word COM` | 構造抽出を優先する |
| `rtf` / `trf` | `Word COM` | `.docx` 正規化後に Word 系として扱う |
| `pptx` | 別途検討 | 現時点では詳細未確定 |
| `pdf` | 別途検討 | 現時点では詳細未確定 |
| `bagles` | 簡易パーサ | 定義書キーワードで区切ってコードブロック化 |
| `text` | そのまま読み込み | エンコーディングフォールバックあり |

設計上の位置づけ:

- ただし、`Docs / Docx / Word` と `RTF / TRF` は優先検討対象であり、Speaker Deck `RAG x FINDY` の考え方を踏まえ、直接 Markdown へ潰す前に構造を取るルートを優先的に検討する
- 特に Word 系では、見出し、段落、表、画像、図形、テキストボックス、アンカー情報などを中間表現として保持してから最終 Markdown を再構成する方針を第一候補とする
- 画像は Markdown タグ検出ベースで後処理するより、構造抽出時点で `図`, `画像`, `shape`, `inline shape` として扱う方向を優先する
- 表は最終的に Markdown テーブルのまま渡すことを前提とせず、項目ラベル付きテキストや半構造化テキストへ変換する方向を優先候補とする

### 3.5 Phase 5: 品質判定 & 構造化

- この段階では、形式ごとに意味を壊していないかを確認しながら整形する
- Word 系では、特に制約事項、条件、表、図の説明が誤読されていないかを重視する
- 見出しが失われていないか、`1つの表 + その前後の説明文` が 1 つの意味まとまりとして保てているかを確認する

### 3.6 Phase 6: Dify 投入向け整形 & 出力

設計方針:

- 当面は Dify `1.13.0` の UI で手動登録する前提のため、最終成果物は `Markdown (.md)` の document を基本とする
- ここでの主眼は「事前に完全なチャンク成果物を作って渡す」ことではなく、「Dify に渡した後、Dify 標準 chunking がうまく機能する形に整形する」ことに置く
- 基本単位は `1ファイル = 1 document` とし、Dify 側で chunk へ分割させる
- したがって、最終出力は `見出し構造`, `項目ラベル`, `説明文の前後関係` を保ちつつ、Dify が自然に分割しやすい構造を優先する
- 見出しは Dify の自動チャンキングの区切りヒントとして保持または必要に応じて付与する
- 表は Markdown テーブル記法のまま渡すことを基本とせず、記号過多による誤分割を避けるため、項目ラベル付きテキストや半構造化テキストへ変換する
- 1 つの意味まとまりは、当面 `1つの表 + その前後の説明文` を基本単位として整形する
- `1チャンク = 1ファイル` のような事前チャンク成果物は、現時点では確定仕様としない
- メタデータの持ち方や追跡情報の定義は、Dify UI での入力方法を踏まえて別途検討する

Word 系の優先方針:

- `Docs / Docx / Word` を最優先の検討対象とする
- Word 系は、ワークフロー図、表、画像、テキストボックス、浮動オブジェクトで構造が崩れやすいため、直接 Markdown 化より `構造抽出 -> 中間表現 -> 再構成` を優先する
- 最終 Markdown は、完全な自然文への書き下しよりも、見出しや項目ラベルを残した半構造化テキストを優先候補とする

RTF / TRF の方針:

- `RTF / TRF` は、まず Word へ正規化して Word 系の方針へ合流させる
- 合流できない場合のみ、補助的なフォールバックを検討する

## 4. BAGLES 定義書の扱い

### 4.1 現在の前提

- 提供済みサンプルでは `.bik` `.bca` `.bci` `.bpg` の 4 拡張子を BAGLES として扱う
- 現行 Markdown 変換は、高度な表構造パースではなく「定義書キーワードで区切って Markdown 化する簡易ルート」である
- 検証用出力は `work/_validation` 配下に存在し、BAGLES の基本通過ルートは確認済み

### 4.2 将来拡張候補

- 富士通ツールからのテキスト / HTML / CSV / PDF エクスポートを優先する
- 直接解析が必要な場合は、定義書種別ごとに専用パーサを追加する
- 大型定義書では「定義書単位」や「モジュール単位」での意味分割を追加する

## 5. 実装上の課題と整理事項

1. BAGLES の拡張子判定がソース上で一元化されていない
2. `.trf` は設計対象に追加したが、実データでの変換品質確認は未実施
3. Phase 3 の分割はまだ意味分割ではなく、行単位分割が主である
4. Word / RTF の構造保持抽出は優先検討対象だが、細部仕様は未確定である
5. 図や画像の扱いは、Markdown 変換後のタグ処理ではなく、構造抽出時点でどう保持するかを詰める必要がある
6. OpenAI 側はモデル更新が早いため、設計書に特定モデル名をハードコードし続けない運用が必要
7. OpenAI の Responses API へ移行する場合は、現行の Chat Completions 互換実装との差分整理が必要
8. Dify の自動チャンキング前提で、見出しや改行の入れ方を実データに近いサンプルで継続調整する必要がある
9. 各形式の詳細な処理仕様は、拡張子別メモで論点を詰めてから確定する

---

## 6. 次のステップ

1. `Docs / Docx / Word` の代表サンプルを基に、壊してはいけない構造を整理する
2. Word 系の見出し付与 / 保持ルールを決める
3. 表を Markdown テーブルではなく、どの項目ラベル付きテキスト形式へ変換するかを決める
4. `1つの表 + その前後の説明文` を 1 つの意味まとまりとして保てる 1 document 内構成を決める
5. `15MB` 超文書の物理分割単位を、章 / 節 / 付録などの観点で決める
6. `RTF / TRF` を Word へ正規化して同一方針へ合流できるかを検証する
7. その後に `Excel` のレコード / 表 / シート単位の扱いを整理する
8. 最後に `BAGLES` の定義書単位と項目単位の扱いを整理する
