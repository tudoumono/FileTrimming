# BAGLES 定義書メモ

作成日: 260311 203032
更新日: 260311 214114

## 1. 結論

- 現行実装の正本は [詳細設計書 v2](../ドキュメント処理パイプライン詳細設計書_v2.md) とし、このメモでは BAGLES 固有の前提、製品理解、論点、将来拡張候補を整理する
- このプロジェクトで会話上に出てくる `Bugless` という呼び方は、富士通の `BAGLES / BAGLESII / Open BAGLES` 系列を指すものとして扱う
  - これは本プロジェクト上の整理であり、正式な製品表記は一次情報に合わせて `BAGLES` を使う
- BAGLES は、一般的な Word 文書や Excel 台帳ではなく、`日本語・表形式の業務仕様書から業務アプリケーションを自動生成する` という思想の開発効率化ツール系列として理解するのが重要
- そのため、BAGLES 定義書は単なる「表っぽい文書」ではなく、`項番`, `区分`, `項目`, `条件`, `エラーコード`, `用語名`, `形式`, `モジュール`, `入出力項目` などが意味を持つ半構造化仕様資産として扱うべき
- 現時点の提供物では `.bik` 8件、`.bca` 1件、`.bci` 1件、`.bpg` 1件の計 11 件を BAGLES として扱う
- 現行 Markdown 変換は、定義書キーワードで区切る簡易パーサを本流とする
- ただし今後の議論では、BAGLES を Word や Excel と同じ感覚で自然文化しすぎず、`定義書の型` と `項目の関係` を残すことを重視する

## 2. BAGLES とは何か

### 2.1 製品の位置づけ

- 富士通の一次情報では、`BAGLES` は `Business Application Generator's Library for Extensive Support` の略とされている
- 1983年に BAGLES、1990年に機能拡張版の `BAGLESII`、2007年にオープンプラットフォーム向けの `Open BAGLES` が提供されている
- 富士通の説明では、`業務用語で記述した表形式の業務仕様書` から、プログラムや管理ドキュメント群を自動生成することが特長である
- 主な適用先は、金融機関を中心とした大規模基幹系システムである

### 2.2 このプロジェクトで重要な理解

- BAGLES 定義書は、人が読む説明資料であると同時に、開発資産生成の元になる仕様資産である
- したがって、1セルや1行が単なる見た目ではなく、`ロジック条件`, `エラーコード`, `データ項目定義`, `業務用語`, `モジュール入出力` などに対応している可能性が高い
- 自然文へ書き下しすぎると読みやすくはなるが、元の定義粒度や列の対応関係を壊しやすい
- RAG 用前処理では、`文章化のしやすさ` より `定義の意味を壊さないこと` を優先した方が安全

## 3. 富士通一次情報から読み取れること

### 3.1 Open BAGLES 販売開始リリースから分かること

- Open BAGLES は、BAGLES 系列の特徴を継承しつつ、Linux / Solaris / Windows へ展開したオープンプラットフォーム向け製品として紹介されている
- 富士通は、業務仕様書からプログラムと管理ドキュメント群を自動生成し、COBOL 開発比で生産性向上をうたっている
- また、`業務仕様書とプログラムの完全一致`, `標準化`, `履歴の可視化`, `メンテナンス性` を前面に出している

この点から、RAG 観点で重要なのは次の通り。

- 仕様書はプログラムと強く結びついている
- 仕様書の表現は、自由文より定型記述が中心である
- `修正履歴`, `定義書間の対応`, `トレーサビリティ` が重要資産である

### 3.2 Open BAGLES 紹介資料 PDF から分かること

- 紹介資料では、BAGLESII の基本機能を `日本語・表形式の業務仕様書を入力として COBOL ソースプログラムを生成するもの` と説明している
- 同資料の例では、以下のような定義書名が確認できる
  - `チェック更新定義書`
  - `業務用語定義書`
  - `コード定義書`
  - `フォーマット定義書`
  - `ロジック定義書`
  - `インタフェース定義書`
  - `履歴定義書`
- 定義書サンプルには、`項番`, `区分`, `項目`, `更新内容`, `チェック内容`, `エラーコード`, `条件1-3`, `備考` のような列が見える
- 別の例では、`モジュール`, `処理`, `条件`, `入出力項目`, `GRP`, `レコード番号` なども確認できる

この点から、BAGLES 定義書は少なくとも次の性質を持つと考えるのが自然。

- 文書というより `表ベースの仕様言語`
- 定義書種別ごとに列の意味が異なる
- `項番` や `区分` が論理順序や識別子として重要
- 同じ業務対象が複数の定義書にまたがって定義される
- エラーコード、条件、用語、データ項目、インタフェースが分離されている

## 4. このプロジェクトでの扱い

### 4.1 現在の提供物

- 現在の提供サンプルでは `.bik`, `.bca`, `.bci`, `.bpg` を BAGLES として扱っている
- これらの拡張子の意味や境界は、現時点では本プロジェクトの実サンプル観察に基づく運用上の整理である
- 富士通の一次情報からこれらの拡張子そのものを裏づける記述は、現時点では確認していない
- したがって、`BAGLES 製品理解` と `今回受領しているファイル拡張子群` は区別して扱う必要がある

### 4.2 現行実装の位置づけ

- 現行実装は、定義書キーワードで区切って Markdown 化する簡易ルートを本流としている
- これは「まず通す」ための暫定ルートとしては有効だが、BAGLES の定義関係や列意味までは十分に保持できない
- 特に `チェック更新定義書`, `ロジック定義書`, `インタフェース定義書` のような定義書では、単純な自然文化やコードブロック化で意味を壊しやすい

## 5. なぜ BAGLES が難しいのか

### 5.1 Word / Excel とは違う難しさ

- Word のように見出しと段落を追えばよいわけではない
- Excel のように単純な表データとして見ても不十分で、`表そのものが仕様言語` になっている
- 列名や項番が意味論の一部であり、列の順番が崩れると解釈を誤る可能性がある
- 1つの定義書だけで意味が完結せず、業務用語定義書とコード定義書、ロジック定義書とインタフェース定義書のように、定義が分散している

### 5.2 RAG で特に危険なこと

- 1行を自然文へ完全に書き換えると、元の列対応が失われる
- 定義書種別を落としてしまうと、同じ語でも意味が変わる
- `項番` を捨てると、順序や参照関係が追えなくなる
- `コード`, `条件`, `形式`, `入出力項目` を雑に連結すると誤読しやすい
- 影響調査用途では、原本のどこに書いてあったかの追跡性が特に重要

## 6. 今後の議論で重視すべき観点

### 6.1 最低限残すべき情報

- 元ファイル名
- 定義書種別
- 項番
- 区分
- 項目名または用語名
- 条件
- エラーコード
- データ形式
- モジュール名やプログラム名
- 入出力項目
- 可能ならページ番号や表番号

### 6.2 Dify 向けに考える時の基本方針

- まず `Dify に取り込めること`
- 次に `Dify UI で扱えること`
- その上で、BAGLES は完全な自然文より `項目ラベル付きの半構造化テキスト` の方が適する可能性が高い
- さらに、定義書種別ごとにチャンク単位を変える方が自然な可能性が高い

### 6.3 現時点の第一候補

- `1ファイル = 1 document` の原則は Dify 側に合わせる
- その document の内部では、定義書種別を明示しつつ、項番単位または小さな定義ブロック単位で意味が追える構造へ整形する
- ただし、Dify に渡す前に JSON チャンクを完成させるのではなく、Dify 標準の chunking が壊しにくい Markdown を作る方針を基本にする

## 7. 定義書種別ごとの見え方

以下は、一次情報と現行サンプルの両方から見て、今後の整理で重要になる種別。

- `チェック更新定義書`
  - チェック条件、更新内容、エラーコードが中心
  - エラーコード検索や入力条件検索と相性が良い

- `業務用語定義書`
  - 用語と意味の対応が中心
  - 用語辞書的な RAG に向く

- `コード定義書`
  - コード値と名称の対応が中心
  - 半構造化テキスト化と相性が良い

- `フォーマット定義書`
  - 項目の階層、形式、データ構造が中心
  - 桁数や型情報を落とさないことが重要

- `ロジック定義書`
  - 条件、処理、モジュール単位の流れが中心
  - 単純な表保持だけではなく、条件のまとまりも必要

- `インタフェース定義書`
  - 入出力項目、GRP、プログラム単位の対応が中心
  - 項目群の境界と入出力方向を落とさないことが重要

- `履歴定義書`
  - 修正番号ごとの履歴管理に関わる
  - 影響調査や変更追跡で重要

## 8. 再確認しやすい論点

- `.bik / .bca / .bci / .bpg` を同一ルートで扱って問題ないか
- 実サンプルの各拡張子は、どの定義書種別と対応しやすいか
- 現行の簡易パーサだけで、定義書の区切りと本文が十分読めるか
- 公式ツールで一括エクスポートできるか
- 定義書種別ごとの共通レイアウトを安定してパースできるか
- 項番、区分、条件列、コード列を壊さずに Markdown 化できるか
- 影響調査用途に必要な追跡性を残せるか
- OCR を最終手段にした場合の品質が足りるか

## 9. 試験時の確認項目

- 実サンプルをテキストエディタで読めるか
- 定義書種別を安定して判定できるか
- 項番、区分、項目名、条件、コードが落ちずに抽出できるか
- 現行の簡易パーサで定義書見出しと本文境界が崩れていないか
- 公式ツールから一括エクスポートできるか
- 定義書種別ごとの共通レイアウトを抽出できるか
- 影響調査に必要な `元ファイル名`, `ページ`, `表位置` などをどこまで残せるか
- Dify に投入した後、質問に対して誤読なく引けるか

## 10. 今後の方針メモ

- BAGLES は、Word や Excel の後で検討する対象ではあるが、`独特な定義書文化` を理解せずに処理方式を決めると失敗しやすい
- そのため、少なくともこのメモの製品理解と定義書種別理解は、今後の議論の前提として固定しておく
- 実装方針としては、`エクスポート優先`, `定義書種別ごとの扱い分離`, `自然文化しすぎない`, `追跡性を残す` を軸にする
